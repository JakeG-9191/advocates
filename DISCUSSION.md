## Discussion of Changes
- Major changes included adding a separate database connection, mostly out of necessity given Docker's unwillingness to work on my personal Windows machine. After configuring the database and seeding, I focused heavily on adding types, removing client facing logs, ensuring routes were able to return data, and on the display of data itself in the updates advocates page. Within the advocates page, we are now able to see a complete list of current providers, the list is sortable, filters via a compound search that includes most core fields within the table, includes pagination, includes a reset flow, includes an error state to quickly inform the user of any issue with data availability, and includes an overhauled UX and applied styles. These styles focus on a cleaner UX to reduce the overall size of each row, including only showing a slice of specializations that the advocate works in, plus makes the table more readable in general, includes formatting for phone numbers if available, and includes aria labels on a number of interactive elements.


## Discussion of Next Steps
- Given additional time, my primary focus would have been to offload sort, filter and pagination logic to the server and add more seed data to prove out the performance changes, including optimized column updates such as ensuring that ambagious fields such as phone number are stored in a more friendly format and that indexes worked as intended. While it can be argued that the UX is better to have some portion of the advocates list loaded as soon as the page is available to serve and to have the client handle sorting, pagination, filtering, etc., this is only manageable in small settings where we have less than a few thousands records to work with, anything larger and we should rely more heavily on the backend and data persistence layer to handle that functionality.
- Additionally, the styling would have seen more work, to ensure better mobile support and to ensure more stable rendering structures as the table is sorted, searched, or inline components are interacted with, such as the list of specialties.
- Another area focused on styling would be to implement both a light and dark mode, where automatic preferences can be set by the users browser, and where these changes can be made persistent
